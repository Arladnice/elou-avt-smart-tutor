## 🔍 3. Стандарты качества кода и Code Review (Code Quality & Review Standards)
Для исключения неструктурированного кода (inline-стили, разбросанные по файлам fetch-запросы, хардкод конфигураций) агент ОБЯЗАН придерживаться стандартов качества кода при любом редактировании или создании UI:
1. **Запрет inline-стилей (`style={{ ... }}`)**:
   - Все стили разметки должны быть реализованы через styled-components или CSS-классы.
   - Использование inline-стилей допускается исключительно для динамически вычисляемых во время выполнения значений (например, динамическая позиция, вычисляемая ширина прогресс-бара).
2. **Вынос Styled Components**:
   - Не загромождайте основные файлы React-компонентов styled-определениями.
   - Выносите все styled-компоненты в отдельный файл стилей `*.styles.ts` в той же папке, экспортируя их и импортируя в основной компонент.
3. **Вынос API-запросов (Fetch/Axios)**:
   - Запрещено прописывать прямые вызовы `fetch('http://localhost:8000/...')` в UI-компонентах.
   - Все запросы должны быть вынесены в централизованный сервис-клиент (например, `src/services/api.ts`). Компоненты должны вызывать методы этого сервиса.
4. **Вынос конфигураций и хелперов**:
   - Конфигурации таблиц (например, `columns` для Ant Design `Table`), наборы опций и хелперы форматирования данных должны быть вынесены в companion-файл `*.config.ts` / `*.config.tsx`.
5. **Запрет использования `!important`**:
   - Категорически запрещено использовать директиву `!important` в стилях (CSS / Styled Components).
   - Для переопределения стилей сторонних библиотек (например, Ant Design) используйте увеличение специфичности селекторов с помощью символов амперсанда (`&&` или `&&&` в Styled Components).

## 🐍 4. Стандарты бэкенда (Python / FastAPI)
Все файлы в `backend/` ОБЯЗАНЫ придерживаться следующих правил:

1. **Модульная структура (запрет монолитов)**:
   - Файлы Python не должны превышать **300 строк**. Если файл растёт — разбить на модули.
   - Целевая структура `backend/`:
     ```
     backend/
     ├── main.py              # Только app = FastAPI(), подключение роутеров, lifespan
     ├── routes/              # FastAPI роутеры (APIRouter)
     │   ├── auth.py          # /api/auth/*
     │   ├── sessions.py      # /api/sessions/*
     │   └── ws.py            # WebSocket обработчик /ws
     ├── services/            # Бизнес-логика
     │   ├── connection_manager.py  # ConnectionManager
     │   └── simulation_loop.py     # Фоновый цикл симуляции
     ├── models/              # Pydantic-модели запросов/ответов
     │   └── schemas.py
     ├── db/                  # Работа с БД
     │   ├── database.py      # init_db(), get_connection()
     │   └── queries.py       # SQL-запросы и функции доступа к данным
     └── utils/               # Утилиты
         ├── security.py      # calculate_integrity_hash(), log_audit_event()
         └── helpers.py       # random_id() и прочее
     ```
2. **Pydantic-модели**:
   - Все Request/Response тела — через `pydantic.BaseModel` с валидацией полей. Запрет `dict` или `Any` в аннотациях типов ответов эндпоинтов.
   - Модели хранятся в `backend/models/schemas.py`, не в файлах роутеров.
3. **Обработка ошибок**:
   - Все эндпоинты используют `fastapi.HTTPException` с конкретным `status_code` и `detail`. Запрет голых `raise Exception(...)`.
   - Для WebSocket — обёртка `try/except WebSocketDisconnect` обязательна.
4. **Docstrings**:
   - Каждый `@router.get/post/put/delete` ОБЯЗАН иметь docstring, описывающий назначение, параметры и формат ответа. FastAPI использует docstrings для авто-генерации OpenAPI-документации.
5. **Логирование**:
   - Использовать модуль `logging` с уровнями `logger.info()`, `logger.warning()`, `logger.error()`. Запрет `print()` в продакшен-коде.
6. **Безопасность**:
   - Секреты (соли, токены, пароли БД) — через `os.environ.get(...)`, с fallback только для dev-режима. Запрет хардкода секретов в коде.
   - CORS: Для продакшена заменить `allow_origins=["*"]` на конкретные домены.
7. **Импорты**:
   - Запрет `import random` внутри функций. Все импорты — в начале файла.
   - Запрет `sys.path.append(...)` в продакшен-коде. Использовать пакетную установку (`pip install -e .`).

## 🧠 5. Стандарты AI/ML-модулей (ai_core/)
Все файлы в `ai_core/` ОБЯЗАНЫ придерживаться следующих правил:

1. **Разделение Train и Inference**:
   - Код обучения (train loop, DataLoader, optimizer) — в отдельном скрипте `train.py`.
   - Код инференса (загрузка модели, predict) — в отдельном модуле (например, `predictive_engine.py`).
   - Один файл НЕ должен содержать и обучение, и инференс.
2. **Воспроизводимость**:
   - В начале обучения ОБЯЗАТЕЛЬНО фиксировать сиды: `random.seed(SEED)`, `np.random.seed(SEED)`, `torch.manual_seed(SEED)`.
   - Значение SEED — через конфигурацию, не хардкод.
3. **Конфигурации и гиперпараметры**:
   - Все гиперпараметры (learning_rate, epochs, batch_size, hidden_dim, num_layers) — в отдельном файле `config.py` или `config.yaml`. Запрет хардкода внутри тела функций.
   - Константы нормирования (`SCALER_MIN`, `SCALER_MAX`, `OUT_MIN`, `OUT_MAX`) — в `config.py`, не в теле модуля инференса.
4. **Метрики и логирование обучения**:
   - Каждый epoch ОБЯЗАН логировать: epoch номер, train loss, val loss (если есть), MAE/R² на валидации.
   - Запрет «тихого» обучения без вывода метрик.
5. **Экспорт модели (ONNX)**:
   - После `torch.onnx.export()` ОБЯЗАТЕЛЬНО выполнить smoke-test: `ort.InferenceSession(path)` + инференс на dummy-данных.
   - Логировать размер экспортированной модели.
6. **Docstrings**:
   - Каждый класс и публичный метод ОБЯЗАН иметь docstring с описанием: назначение, формат входа/выхода, единицы измерения.
7. **Технологические константы**:
   - Все аварийные пороги и физические пределы (T > 380°C → авария, P > 0.6 МПа → авария) должны быть определены как именованные константы, а не magic numbers.
   - Пример: `FURNACE_TEMP_CRITICAL = 380.0  # °C, предел аварийного останова`.

## ⚙️ 6. Стандарты физической модели (simulator/)
Все файлы в `simulator/` ОБЯЗАНЫ придерживаться следующих правил:

1. **Физические пределы (clamping)**:
   - Все вычисленные значения температуры, давления, уровня ОБЯЗАНЫ проходить через `np.clip()` или `max(min(...))`.
   - Запрет отрицательных значений давления, температуры ниже абсолютного нуля, уровня вне [0, 100]%.
   - Допустимые диапазоны (из техрегламента):
     - `furnaceTemp`: 20.0 — 600.0 °C
     - `columnPres`: 0.02 — 1.5 МПа
     - `columnLevel`: 0.0 — 100.0 %
2. **Docstrings с физическим смыслом**:
   - Каждый метод модели ОБЯЗАН документировать:
     - Какой физический процесс он моделирует.
     - Единицы измерения входных/выходных параметров (°C, МПа, %).
     - Ссылку на пункт техрегламента, если применимо.
3. **Сценарии**:
   - Список поддерживаемых сценариев: `startup`, `shutdown`, `column_shutdown`, `overpressure_relief`, `recirculation`.
   - Для каждого сценария метод `reset(scenario_id)` ОБЯЗАН задокументировать начальные условия.
4. **Неисправности (дефекты)**:
   - Каждый дефект (`pump_fail`, `coil_overheat`, `valve_jam`) ОБЯЗАН иметь docstring с описанием физического эффекта.
   - При добавлении нового дефекта — ОБЯЗАТЕЛЬНО добавить его в `ConnectionManager` и в `DEFECT_NAMES_RU` для локализации.

## 🧪 7. Стандарты тестирования
Агент ОБЯЗАН соблюдать следующие требования при работе с тестами:

1. **Backend (pytest)**:
   - Каждый новый REST-эндпоинт ОБЯЗАН иметь хотя бы один тест в `backend/tests/`.
   - Тесты используют `httpx.AsyncClient` с `app` из `main.py`.
   - Запуск: `cd backend && python -m pytest tests/ -v`.
2. **Simulator (pytest)**:
   - Unit-тесты для каждого из 5 сценариев: `startup`, `shutdown`, `column_shutdown`, `overpressure_relief`, `recirculation`.
   - Тесты проверяют: начальные условия, корректность шага физики, граничные условия (аварийный останов при T > 380°C).
3. **AI/ML**:
   - Smoke-тест ONNX-модели: загрузка + инференс на dummy-данных (нулевой вектор) без ошибок.
   - Проверка, что выход `predict_risk()` возвращает `risk` в диапазоне [0, 100].
4. **Frontend (TypeScript)**:
   - Перед любым мержем или завершением задачи: `cd frontend && npx tsc --noEmit`.
   - При критических изменениях: `cd frontend && npm run build` (должен пройти без ошибок).
