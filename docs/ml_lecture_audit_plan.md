## Детальный анализ пробелов

### 🔴 Критические пробелы (влияют на оценку жюри)

#### GAP-1: Нет Baseline и сравнения с ним
**Проблема:** Спикер многократно подчеркивал обязательность baseline. У нас LSTM используется «в вакууме» — без доказательства, что она лучше простого порогового правила.

**Что есть:** В `ai_core/config.py` (строки 37-53) уже определены все пороги из техрегламента (`FURNACE_TEMP_CRITICAL = 380.0`, `COLUMN_PRES_WARNING = 0.40` и т.д.).

**Что нужно сделать:**
1. В `train.py` после обучения: на validation set прогнать **baseline-предиктор** (пороговое правило: если `T > 350°C` → risk > 70%, иначе low risk).
2. Для того же validation set прогнать LSTM.
3. Бинаризировать `risk > 50%` → «авария» и посчитать **Recall, Precision, F1, PR-AUC** для обоих подходов.
4. Посчитать **Lead Time** — за сколько шагов до реальной аварии модель предупредила.
5. Сохранить результаты в `ai_core/evaluation_report.md`.

**Файлы:** `ai_core/train.py`, новый `ai_core/evaluate.py`

---

#### GAP-2: Нет классификационных метрик (только MSE/MAE)
**Проблема:** `train.py` логирует только MAE по температуре, давлению и уровню. Но бизнес-задача — **предсказание аварии**, а это задача классификации (авария / не авария).

**Что нужно сделать:**
1. Создать `ai_core/evaluate.py` — модуль честной оценки.
2. На тестовом наборе конвертировать прогнозы T/P/L в бинарный risk (пороги из `config.py`).
3. Рассчитать Recall, Precision, PR-AUC, F1-score.
4. Построить PR-кривую и сохранить как изображение для презентации.

**Файлы:** Новый `ai_core/evaluate.py`

---

#### GAP-3: Нет Test set (только Train/Val)
**Проблема:** В `train.py:81` данные делятся на 80% train и 20% val. Отдельного test set нет. Спикер прямо запретил использовать val для финальной оценки.

**Что нужно сделать:**
1. Изменить разбиение: 70% train / 15% val / 15% test (по `TRAIN_VAL_SPLIT`).
2. Добавить в `config.py`: `TRAIN_SPLIT = 0.7`, `VAL_SPLIT = 0.15` (test = остаток).
3. Финальные метрики считать **только** на test set.

**Файлы:** `ai_core/config.py`, `ai_core/train.py`

---

#### GAP-4: RAG-защитное правило отсутствует
**Проблема:** `generate_rag_fallback()` в `ai_chat_service.py` (строки 173-346) всегда генерирует ответ, даже если вопрос нерелевантен. `vector_store.search()` на строке 117 возвращает документы даже при `sim > 0.01` — порог слишком низкий.

> *Спикер: «Хорошая система, если не находит контекст или найденного контекста недостаточно, должна сообщить об этом»*

**Что нужно сделать:**
1. В `vector_store.py` повысить минимальный порог релевантности.
2. В `ai_chat_service.py` добавить проверку: если ни один keyword не совпал и `vector_store.search()` вернул результаты с низким score — вернуть «Информация по данному запросу не найдена в базе знаний. Уточните вопрос или обратитесь к инструктору».
3. Передать это и в system prompt для LLM.

**Файлы:** `backend/services/vector_store.py`, `backend/services/ai_chat_service.py`

### 🟡 Средние пробелы (улучшают качество, но не блокируют)

#### GAP-5: Предписывающие формулировки (а не директивные)

**Рекомендация спикера:** «Рекомендуется снизить нагрузку и провести осмотр» — предписывающий формат, оставляющий решение за человеком.

**Что нужно сделать:**

**Что нужно сделать:**

2. На бэкенде — эндпоинт `POST /api/alarm-feedback` с записью в журнал аудита.

#### GAP-7: Утечка данных между сессиями при генерации датасета
**Проблема:** `data_generator.py` генерирует данные сессиями по 100-300 шагов. Затем `train.py` берёт весь CSV и нарезает окна `X[i:i+30]` → `y[i+30+15]`. Если две смежные сессии оказались в разных выборках (train/val), окно может «перешагнуть» через границу сессий, создав нереалистичный переход.

**Что нужно сделать:**
1. В `data_generator.py` добавить маркер `session_id` в CSV.
2. В `train.py` при нарезке окон не пересекать границы сессий.
3. При разбиении Train/Val/Test — группировать по `session_id` (целые сессии попадают в один сплит).

**Файлы:** `ai_core/data_generator.py`, `ai_core/train.py`

---

#### GAP-8: Чанкинг RAG примитивен
**Проблема:** `vector_store.py:47` разбивает текст по `\n\n`. Спикер подчёркивал, что *«чанкинг — это feature engineering для RAG»* и нельзя *«рвать процедуру посередине»*.

**Что нужно сделать:**
1. Разбивать по заголовкам markdown (`## ...`, `### ...`).
2. Добавлять метаданные: `section`, `equipment_type`, `regulation_clause`.
3. Поля метаданных использовать для фильтрации при поиске.

**Файлы:** `backend/services/vector_store.py`
