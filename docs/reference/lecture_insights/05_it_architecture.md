## Основные тезисы

### 1. Три вида архитектуры
| Вид | Фокус | Аудитория | Инструменты |
|---|---|---|---|
| **Software Architect** | Качество кода, паттерны | Разработчики | IDE, Git, UML |
| **Solution Architect** | Выбор варианта реализации, интеграция в ландшафт | Все стейкхолдеры | Visio, квадраты и линии |
| **Enterprise Architect** | Стратегия, ландшафт организации | Топ-менеджмент | PowerPoint (!), Archimate |

### 2. Solution Architect — ключевая роль для нашего проекта
- Ведёт ландшафт из состояния A в состояние Б, закрывая потребности бизнеса
- Выбирает путь реализации: быстро/дёшево/с долгами vs дорого/долго/без долгов
- Объясняет разработчикам паттерны, эксплуатации — мониторинг/SLA, бизнесу — ценность
- Связующее звено между всеми участниками проекта

### 3. Представление архитектуры — модель 4+1
Нужно рассказать решение с разных точек зрения:
1. **Логическое** — для пользователей (функциональность)
2. **Разработки** — для разработчиков (структура кода, модули)
3. **Процессное** — для интеграторов (производительность, масштабируемость)
4. **Физическое** — для инженеров (сервера, сети, топология)
5. **Сценарии (Use Cases)** — объединяющий слой

### 4. Три варианта реализации любого решения
- **Вариант 1:** Быстро, дёшево, с техдолгом, высокие риски, плохая масштабируемость
- **Вариант 2:** Умеренный компромисс, средние затраты, приемлемые сроки
- **Вариант 3:** Дорого, долго, без долгов, масштабируемо, но нужен бюджет

> *«Мы всегда стараемся посмотреть как можно большее количество вариантов»*

### 5. Archimate — три слоя описания
- **Технологический:** сервера, сети, устройства
- **Прикладной:** приложения, объекты данных, функциональность
- **Бизнес-слой:** процессы, сервисы, ценности для пользователя

## Применение к нашему проекту КТК ЭЛОУ-АВТ

### Архитектурное представление по модели 4+1

#### Логическое (для жюри/бизнеса)
```
┌────────────────────────────────────────────┐
│          КТК ЭЛОУ-АВТ Smart Tutor         │
├──────────┬──────────┬──────────┬───────────┤
│  АРМ     │  АРМ     │ ИИ-      │ Физич.   │
│  Оператора│ Инструктора│ Модуль  │ Модель   │
│  (SCADA) │  (Панель)│ (LSTM)  │ (Симул.) │
└──────────┴──────────┴──────────┴───────────┘
```

#### Разработки (для нашей команды)
```
frontend/    → React 18 + TypeScript + Ant Design + styled-components
backend/     → FastAPI + WebSocket + SQLite
ai_core/     → PyTorch LSTM → ONNX Runtime
simulator/   → NumPy физическая модель
```

#### Процессное (производительность)
- WebSocket: обновление мнемосхемы 1 Hz
- ONNX-инференс: < 200 мс
- SQLite: аудит-лог без блокировок

#### Физическое (деплой)
- Один ноутбук (демо-режим): `docker-compose up`
- Продакшен: раздельные контейнеры на сервере

### Выбор варианта реализации (для пояснительного документа)
| Критерий | Вариант 1: Коленка | **Вариант 2: Smart-MVP (наш)** | Вариант 3: Enterprise |
|---|---|---|---|
| Сроки | 2 недели | **6 недель** | 6+ месяцев |
| Стоимость | ~200K₽ | **~2.4M₽** | ~10M₽+ |
| ИИ | Нет | **LSTM + локальная БЗ** | LLM + MLOps pipeline |
| Масштабируемость | Нет | **JSON-конфиг** | Полный конфигуратор |
| Техдолг | Высокий | **Умеренный** | Минимальный |
