## Основные тезисы

### 1. Пирамида требований (по BABOK)
```
Потребность (проблема)
  └→ Бизнес-требования (цель бизнеса, SMART)
       └→ Требования заинтересованных сторон (User Stories)
            └→ Функциональные требования (что делает система)
            └→ Нефункциональные требования (как хорошо делает)
                 └→ Переходные требования (миграция, обучение)
```

### 2. Бизнес-требования — формулируются через цель, не через решение
- Не «разработать портал», а «повысить долю обученных с 68% до 90% к концу года»
- Всегда SMART: конкретное, измеримое, достижимое, целесообразное, ограниченное по времени
- На проекте обычно 3–5, не более 10 бизнес-требований

### 3. User Stories — формат записи требований заинтересованных сторон
- Формула: «Я как [роль] хочу [действие], чтобы [ценность]»
- Последняя часть («чтобы...») — самая важная, показывает ценность
- Одно бизнес-требование порождает множество User Stories

### 4. Функциональные vs нефункциональные требования
- **Функциональные:** «Система должна позволять скачать сертификат» (не «добавить кнопку»!)
- **Нефункциональные:** производительность, надёжность, ИБ, масштабируемость, совместимость
- Правило: если требование не описывает функцию, а определяет качество/ограничение → НФТ

### 5. Переходные требования
- Нужны только один раз при внедрении
- Перенос данных, обучение пользователей, параллельная работа старой и новой системы

## Применение к нашему проекту КТК ЭЛОУ-АВТ

### Бизнес-требования КТК
| # | Бизнес-требование (SMART) |
|---|---|
| БТ-1 | Сократить время подготовки нового оператора ЭЛОУ-АВТ до самостоятельной работы на 30% за счёт тренажёрного обучения |
| БТ-2 | Обеспечить 100%-е прохождение сценариев аварийных ситуаций каждым оператором до допуска к реальной установке |
| БТ-3 | Снизить количество инцидентов на установке ЭЛОУ-АВТ, связанных с человеческим фактором, на 20% в год |

### User Stories для КТК
| Роль | User Story |
|---|---|
| Оператор-стажёр | Я как оператор-стажёр хочу видеть подсказки на мнемосхеме при отклонении параметров, чтобы научиться распознавать аварийные ситуации |
| Оператор-стажёр | Я как оператор-стажёр хочу видеть тренды параметров за последние 30 минут, чтобы развить интуицию к динамике процесса |
| Инструктор | Я как инструктор хочу задавать неисправности в любой момент тренировки, чтобы проверить реакцию оператора на нештатные ситуации |
| Инструктор | Я как инструктор хочу видеть журнал действий оператора с таймстемпами, чтобы оценить скорость и правильность реакции |
| Руководитель цеха | Я как руководитель хочу видеть статистику прохождения сценариев, чтобы принимать решения о допуске к работе |

### Нефункциональные требования КТК
| Категория | Требование |
|---|---|
| Производительность | Обновление показаний мнемосхемы — не реже 1 раза в секунду |
| Производительность | Ответ ИИ-модуля на запрос прогноза — не более 200 мс |
| Надёжность | Система должна корректно восстанавливать состояние тренировки после обрыва WebSocket-соединения |
| ИБ | Запрет НСД к АРМ Инструктора — аутентификация с разделением ролей |
| ИБ | Журнал аудита всех действий с невозможностью подмены результатов |
| Масштабируемость | Архитектура должна позволять добавление новых установок через конфигурационный файл |
