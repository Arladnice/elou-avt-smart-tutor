## Основные тезисы

### 1. Модель Cynefin — определи систему, прежде чем действовать
- **Простые системы:** есть стандартизированный процесс, действуй по инструкции
- **Сложные (комплексные):** нужна экспертиза, можно нанять специалиста
- **Неопределённые:** нет данных, цель есть, путь непонятен → нужны гипотезы
- **Хаос/кризис:** действовать немедленно, неважно правильно или нет

> *«Зачастую мы находимся в беспорядке — не понимаем, в какой системе мы находимся»*

### 2. Цикл ODA (Observe → Orient → Decide → Act)
Правильный подход к решению задач в неопределённости:
1. **Observe** — собрать факты, определить что известно/неизвестно
2. **Orient** — сориентироваться в контексте, сформулировать гипотезы
3. **Decide** — выбрать одну ключевую гипотезу
4. **Act** — пойти в тестирование через прототипы, не сразу в целевое решение

### 3. PreMortem — анализ возможного провала
- Перед запуском эксперимента представить: «наше решение не сработало, почему?»
- Позволяет увидеть риски, слабые места, перегруппироваться
- Спикер рекомендовал: «Когда вы первый раз соберётесь командой и наметите план — подумайте через PreMortem, что может пойти не так»

### 4. ИИ как помощник в цикле ODA
- Можно давать задачу ИИ: «реши проблему по принципу ODA» — подсветит зоны неопределённости
- Верификация остаётся за человеком: «какое решение предлагаем, какую гипотезу причины проверяем, какую метрику успеха поставим»
- Предупреждение: «есть практика — хотя бы неделю без ИИ, чтобы мозг сохранял гибкость»

## Применение к нашему проекту КТК ЭЛОУ-АВТ

### Анализ по Cynefin
Наш кейс находится на стыке **сложной** и **неопределённой** систем:
- Сложная: есть регламенты, техпроцесс известен, физика моделируема
- Неопределённая: нет реальной телеметрии, неясно какой UX будет лучшим для оператора

### Применение ODA к нашему решению
1. **Observe:** Изучили 13 документов + 6 лекций → зафиксировали факты и пробелы
2. **Orient:** Сформулировали гипотезу: LSTM на синтетике + локальная БЗ лучше чем LLM
3. **Decide:** Выбрали Гибридный Smart-MVP
4. **Act:** Тестируем через рабочий прототип, не через громоздкую архитектуру

### PreMortem для нашего решения
Потенциальные провалы:
- Синтетика может не отражать реальную динамику → нужна валидация экспертом
- LSTM может давать высокий risk на нормальных данных → нужна калибровка порогов
- SCADA-панель может быть непонятна жюри → нужен onboarding / guided tour
- Демонстрация на ноутбуке может тормозить → нужна оптимизация сборки
