## Основные тезисы

### 1. Путь зрелости мониторинга
```
Реактивный → Проактивный → Предиктивный
(узнаём от клиента)  (узнаём раньше клиента)  (предсказываем до проблемы)
```

> *«Если вы проблемы узнаёте по звонку разгневанного клиента — вы уже опоздали»*

### 2. Три столпа observability (наблюдаемости)
- **Метрики** — приборная панель: числа во времени (CPU, запросы, время ответа)
- **Логи** — чёрный ящик: неизменяемые записи о событиях
- **Трейсы** — маршрут запроса через микросервисы (где задержка/ошибка)

Когда все три собраны в одном месте и сопоставлены — это **observability**.

### 3. Классификация → группировка → маршрутизация (важнее чем AI!)
> *«Типичная ошибка — сразу пытаться спроектировать AIOps с умной корреляцией, не сделав базовую классификацию событий и маршрутизацию»*

Правильный порядок:
1. **Единый формат входных событий** (source, severity, object, text, tags)
2. **Классификация** (critical/warning/info)  
3. **Дедупликация** (fingerprint → новая проблема или обновление существующей)
4. **Маршрутизация** (куда и кому отправить алерт)
5. **AI-агрегатор** (в самом конце, когда есть база)

### 4. Единый формат сообщений — мост между системами
```json
{
  "source": "zabbix",
  "severity": "warning",
  "object": "server-prod-01",
  "text": "CPU > 95%",
  "tags": ["infrastructure", "cpu"]
}
```
- Приведение критичности к единой шкале **сразу на входе** (адаптером)
- Без указания объекта не получится группировка
- Tags пригодятся для маршрутизации и фильтра в личном кабинете

### 5. Self-service подписки — необходимость, не роскошь
> *«Ручное управление подписками физически не масштабируется»*

- Пользователь сам управляет тем, на что подписан
- Не жёстко привязывать к роли — гибкие настройки

### 6. Пять тезисов на память
1. Мониторинг — это путь зрелости, а не состояние
2. Классификация/группировка важнее AI
3. Self-service подписок — необходимость
4. Единый формат сообщений — мост между системами
5. Безопасность и lifecycle — часть архитектуры с первого дня

### 7. Советы по прототипу
- Начать с 1–2 источников (не пытаться поддержать все сразу)
- Простой self-service лучше идеальной, но неиспользуемой RBAC
- Тестировать модуль за модулем, не дожидаясь полной сборки
- Нагрузочное тестирование: генерировать тысячи событий, найти bottleneck

## Применение к нашему проекту КТК ЭЛОУ-АВТ

### Аналогия мониторинга в нашем КТК
Наш КТК — это, по сути, **система мониторинга технологического процесса**:
- **Метрики** = показания датчиков (температура, давление, уровень)
- **Логи** = журнал событий и действий оператора
- **Трейсы** = цепочка причинно-следственных связей при аварии

### Путь зрелости оператора (аналогия мониторинга)
```
Реактивный оператор      →  Проактивный оператор     →  Предиктивный оператор
(реагирует на аларм)         (видит тренды, замечает      (ИИ предсказывает
                              отклонения до аларма)        риски до отклонений)
```

**Это именно то, чему наш КТК обучает!**

### Классификация алармов в КТК (по аналогии с лекцией)
| Severity | Описание | Пример в ЭЛОУ-АВТ |
|---|---|---|
| **CRITICAL** | Аварийный останов, немедленное действие | T печи > 380°C, P колонны > 0.6 МПа |
| **WARNING** | Предупреждение, требует внимания | T печи > 350°C, уровень куба < 20% |
| **INFO** | Информационное сообщение | Старт сценария, смена режима |

### Единый формат событий КТК
```json
{
  "source": "simulator",
  "severity": "warning",
  "object": "furnace_P1",
  "text": "Температура выхода змеевика превысила 350°C",
  "tags": ["temperature", "furnace"],
  "timestamp": "2026-07-18T10:30:00Z",
  "value": 352.7,
  "threshold": 350.0
}
```

### Маршрутизация алармов в КТК
- CRITICAL → звуковой сигнал + мигающий индикатор на мнемосхеме + запись в журнал
- WARNING → жёлтая подсветка параметра + подсказка от ИИ
- INFO → запись в журнал событий
